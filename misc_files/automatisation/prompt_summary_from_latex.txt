## KR Creation Framework

**CRITICAL**: Every problem type students must solve needs a KR block.

### KR Structure Template

```latex
\begin{KR}{How to [Task Name]}\\
\textbf{Given:} [What information you start with]

\textbf{Goal:} [What you're solving for / proving / calculating]

\textbf{Procedure:}
\begin{enumerate}
    \item [Step 1 with specific action]: [formula or method] (reference Theorem X.Y if applicable)
    \item [Step 2 with decision point]: If [condition], then [action A]; else [action B]
    \item [Step 3 with calculation]: [Specific instruction with example]
    \item [Step 4 with verification]: Check that [condition holds / sanity check]
\end{enumerate}

\textbf{Common mistakes:}
\begin{itemize}
    \item [Mistake 1]: [Description] â†’ [How to avoid]
    \item [Mistake 2]: [Description] â†’ [How to avoid]
\end{itemize}

\textbf{When to use this approach:} [Recognition criteria - how to identify this problem type]
\end{KR}
```

### KR Examples by Domain

**Mathematics**:
```latex
\begin{KR}{How to Prove by Induction}\\
\textbf{Given:} Statement $P(n)$ to prove for all $n \geq n_0$

\textbf{Goal:} Show $P(n)$ is true for all $n \geq n_0$

\textbf{Procedure:}
\begin{enumerate}
    \item \textbf{Base case}: Prove $P(n_0)$ is true directly
    \item \textbf{Inductive hypothesis}: Assume $P(k)$ is true for arbitrary $k \geq n_0$
    \item \textbf{Inductive step}: Prove $P(k+1)$ is true using assumption $P(k)$
    \begin{itemize}
        \item Start with $P(k+1)$ (left side)
        \item Apply inductive hypothesis $P(k)$
        \item Manipulate to show $P(k+1)$ holds
    \end{itemize}
    \item \textbf{Conclusion}: By mathematical induction, $P(n)$ true for all $n \geq n_0$
\end{enumerate}

\textbf{Common mistakes:}
\begin{itemize}
    \item Forgetting base case â†’ Always prove $P(n_0)$ first
    \item Assuming what you're trying to prove â†’ Only assume $P(k)$, prove $P(k+1)$
    \item Incomplete inductive step â†’ Must explicitly show how $P(k)$ implies $P(k+1)$
\end{itemize}

\textbf{When to use:} Statement involves "for all natural numbers $n$" or similar.
\end{KR}
```

**Databases**:
```latex
\begin{KR}{How to Normalize to 3NF}\\
\textbf{Given:} Relation $R$ with attributes and functional dependencies (FDs)

\textbf{Goal:} Decompose $R$ into relations in Third Normal Form (3NF)

\textbf{Procedure:}
\begin{enumerate}
    \item \textbf{Ensure 1NF}: Check all attributes contain only atomic values
    \begin{itemize}
        \item If multi-valued attributes exist, create separate relation
    \end{itemize}
    \item \textbf{Ensure 2NF}: Remove partial dependencies
    \begin{itemize}
        \item Identify composite keys
        \item Find non-key attributes dependent on only part of key
        \item Create separate relation for each partial dependency
    \end{itemize}
    \item \textbf{Ensure 3NF}: Remove transitive dependencies
    \begin{itemize}
        \item Find non-key attributes that determine other non-key attributes
        \item For each transitive dependency $A \to B \to C$, create relation $(A, B)$ and $(B, C)$
    \end{itemize}
    \item \textbf{Verify}: Check all FDs preserved and decomposition is lossless
\end{enumerate}

\textbf{Common mistakes:}
\begin{itemize}
    \item Forgetting to verify lossless join property â†’ Always check can reconstruct original relation
    \item Missing indirect transitive dependencies â†’ Carefully trace all FD chains
    \item Incorrect identification of partial dependencies â†’ Review definition: non-key depends on part of composite key
\end{itemize}

\textbf{When to use:} Asked to "normalize relation to 3NF" or "eliminate redundancy".
\end{KR}
```

**Programming**:
```latex
\begin{KR}{How to Implement Binary Search}\\
\textbf{Given:} Sorted array $arr$ and target value $target$

\textbf{Goal:} Return index of $target$ in $arr$, or -1 if not found

\textbf{Procedure:}
\begin{enumerate}
    \item \textbf{Initialize}: Set $left = 0$, $right = arr.length - 1$
    \item \textbf{Loop while} $left \leq right$:
    \begin{itemize}
        \item Calculate $mid = \lfloor (left + right) / 2 \rfloor$
        \item If $arr[mid] == target$: return $mid$ (found)
        \item If $arr[mid] < target$: set $left = mid + 1$ (search right half)
        \item If $arr[mid] > target$: set $right = mid - 1$ (search left half)
    \end{itemize}
    \item \textbf{If loop exits}: return -1 (not found)
\end{enumerate}

\textbf{Common mistakes:}
\begin{itemize}
    \item Integer overflow in $mid$ calculation â†’ Use $mid = left + (right - left) / 2$
    \item Off-by-one errors â†’ Ensure $left \leq right$ (not $<$) and $mid + 1 / mid - 1$ (not $mid$)
    \item Forgetting array must be sorted â†’ Binary search only works on sorted arrays
\end{itemize}

\textbf{When to use:} Searching sorted array, complexity must be $O(\log n)$.
\end{KR}
```

## Structure Optimization

### Logical Organization (NOT Lecture Order)

Pattern:
```latex
\section{[Thematic Chapter Name]}

% Group related short definitions with \mult{2}
\mult{2}
\begin{definition}{Concept A}\\
[Concise definition]
\end{definition}

\begin{definition}{Concept B}\\
[Concise definition]
\end{definition}
\multend

% Show relationship
\begin{theorem}{How A and B Relate}\\
[Key relationship, comparison, or rule]
\end{theorem}

% Procedure for application
\begin{KR}{When to Use A vs B}\\
[Decision criteria with step-by-step procedure]
\end{KR}

% Concrete application
\begin{example2}{Applying A and B - Medium}\\
[Problem demonstrating both concepts]
\tcblower
[Solution showing decision-making process]
\end{example2}
```

### Layout Principles

**Multi-column grouping** (`\mult{2}...\multend`):
- Use for short related blocks (definitions, concepts, theorems)
- Typical: 2-4 blocks per group
- Balances content across columns

**Single-column content**:
- Long KRs (>10 lines)
- Complex examples
- Detailed explanations
- Wide tables or code

**Column control**:
- `\raggedcolumns` before `\columnbreak` to balance columns
- Strategic `\pagebreak` between major sections
- Reference: See CT2 main file for pattern

### Section Hierarchy

**Appropriate nesting** (max 3 levels):
```
\section{Main Topic}          % Chapter level
  \subsection{Subtopic}        % Major concept
    \subsubsection{Detail}     % Specific aspect
```

**Use `\paragraph{}`** for minor points instead of excessive `\subsubsection{}`

**Merge subsections** if <3 content blocks in section

**Avoid**:
- Single subsection in section (just use paragraphs)
- Deeply nested structure (>3 levels)
- Redundant section titles
